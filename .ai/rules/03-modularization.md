# Modularization Rules

## Code Organization
- Organize code in logical modules with clear responsibilities
- Use meaningful and consistent naming conventions
- Keep functions and classes small and focused (single responsibility)
- Avoid tight coupling between modules

## Reusability
- Extract common functionality into reusable components
- Use composition over inheritance
- Create shared utilities for repeated logic
- Build domain-specific libraries when appropriate

## Maintainability
- Write self-documenting code with clear intent
- Keep modules loosely coupled and highly cohesive
- Avoid circular dependencies
- Use dependency injection for flexibility

## Testing
- Write unit tests for business logic
- Create integration tests for module interactions
- Mock external dependencies in tests
- Aim for high test coverage on critical paths
